# Guard shield tap-churn bubble stick — 2026-06-05

**Status:** FIX SHIPPED (phase 32: Option C — authoritative shield-hold predicate OR-folds the applied input at one chokepoint; fixes GuardOn/GuardOff spam across all release/teardown sites when holding R to grab then shield)  
**Scope:** `port/net/sys/netrollbacksnapshot.c`, `port/net/sys/netrollback.c`, `port/net/sys/netinput.c`

## Symptom

Offline / non-rollback: spam-tap Z/L shield puts bubble up/down on press/release. With rollback active, the fighter body guard animation tracks correctly but the shield **bubble effect** can stay visible between taps.

Soak 2026-06-05: `eff=0x7037852F` frozen for 50+ ticks while fighter `anim` hash churned; zero `effect-repair ok` lines.

Prior logs showed frequent `LOAD_HASH_DRIFT` with matching `figh` but mismatched `eff` (soft-continued as presentational).

## Root cause

Shield sim state (`is_shield`, guard status) lives in the fighter blob; the bubble is a separate effect GObj repaired only on snapshot load. Orphan/duplicate shield effects survived when:

1. `TryRepairEffectHashForVerify` rebind-only path did not run ensure/prune/dedupe.
2. `LOAD_HASH_DRIFT` resim soft-continue tolerated eff mismatch while sim core matched.
3. No per-tick live cleanup between rollback loads during rapid guard tap churn.
4. **Phase 2:** `PruneStaleShields` keyed off guard *status range* instead of `is_shield` — bubble survived tap-off while body left guard anim.
5. **Phase 3:** Live reconcile skipped during resim (`AfterBattleUpdate` early-out); orphan shields survived when `guard->effect_gobj` decoupled from live effect after vanilla `ProcStopEffect`.
