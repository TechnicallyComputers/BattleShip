# Known Bugs & Resolved Issues

This directory documents significant bugs encountered during the port, their symptoms, root causes, and fixes. When fixing a new bug, add an entry here so future sessions can recognize the same class of issue.

| 2026-06-16 | [overscan_black_border](overscan_black_border_2026-06-16.md) | **FIXED (libultraship + port menu)** — A black border framed the game image on all four sides in every window size and fullscreen (distinct from the 4:3 pillarbox; predates the libultraship integration). Root cause: SSB64 bakes a 10px "title-safe" overscan margin into its own GBI — the default scissor (`sys/rdp.c` `syRdpResetSettings` → `(10,10)..(310,230)`) and every per-camera scissor (`sys/objdisplay.c` `dGCCameraScissor{...}=10`) clip all drawing *and the per-frame framebuffer clear* to the safe area, so the 10px margin is never drawn into or even cleared. A CRT's overscan hid it; presenting the full 320×240 framebuffer 1:1 to a desktop window exposes it. Can't just widen the scissor (would reveal uncleared garbage in the margin). Fix: present-level overscan crop in `Gui::DrawGame` — the `ImGui::Image` blit samples the framebuffer minus its proportional margin (`10/320` x, `10/240` y), stretching the safe area to fill the viewport, reproducing the on-TV image. Resolution-independent, orientation-symmetric, skipped in pixel-perfect mode. CVar-gated (`gAdvancedResolution.Overscan.{Enabled,Horizontal,Vertical}`, default on/10/10) with menu toggle + sliders under Enhancements; takes effect immediately (no reload). Audit hook: "image inset from window" → distinguish 4:3 pillarbox (left/right only) vs this overscan frame (all four sides) vs an ImGui dockspace sizing bug (would also affect menu bar / floating windows). |
| 2026-06-15 | [funchook_macos_arm64_patch_anchor](funchook_macos_arm64_patch_anchor_2026-06-15.md) | **FIXED (build)** — TCC mod funchook inline detours silently failed on Apple Silicon (`mprotect(RWX)` → EINVAL on code-signed `__TEXT`) even with JIT entitlements + maxprot re-sign. The FetchContent patch that rewrites funchook to `mach_vm_protect`/`VM_PROT_COPY` never reached the call sites: its CMake anchors used `\;`, which renders as a literal backslash-semicolon, so `string(REPLACE)` no-op'd and only the (now-dead) helper defs got inserted. Fixed anchors to plain `;` + hardened the idempotency guard to key on the wrapped call site (not the helper name). Scripting/events/symbol-resolution were unaffected; only raw detours. Wipe `build*/_deps/funchook-*` to re-patch. Verified via `workspace/hooktest` (1263 detour frames). |
| 2026-06-10 | [controller_port_routing](controller_port_routing_2026-06-10.md) | **FIXED (libultraship)** — Second gamepad (any type) effectively unusable for local multiplayer: three independent defects. (1) Input-editor device rows derived ImGui IDs from device names → two same-model pads collided (warning + hover cross-talk); fixed with instance-id ID suffix + `#1`/`#2` display ordinals. (2) `RefreshConnectedSDLGamepads` ran on every device add/remove event and re-ignored every pad for ports 2-4, silently reverting user routing; now only never-seen instance ids get defaulted. (3) All pads defaulted to port 1 AND only port 1 got default mappings, so a pad routed to port 2 was dead without a manual "Set Defaults"; now Nth pad auto-assigns to first free port and ports 2-4 seed SDL defaults at init. Diagnostic gotchas: same-model pads share SDL GUIDs (per-model, not per-unit); serials are null under macOS GCController driver; headless SDL probes see zero pads without a pumped runloop |
| 2026-06-10 | [crash_handler_devnull_probe](crash_handler_devnull_probe_2026-06-10.md) | **FIXED — corrects a 2026-06-09 audit item** — Crash handler double-faulted on any game-coroutine crash: `ProbeReadable` wrote probed bytes to /dev/null, whose driver (XNU `nullwrite` / Linux `write_null`) never copies from the source buffer, so the probe reported EVERY address readable. `WalkFPChain` then stepped onto the adjacent coroutine stack's `PROT_NONE` guard page at the chain's base frame → nested SIGBUS inside the handler, original backtrace lost, macOS report blamed `port_watchdog.cpp`. Fingerprint: two `!!!! CRASH` lines in ssb64.log, second always SIGBUS at a page-aligned addr. Fix: probe via self-pipe (pipe writes copyin → genuine EFAULT), fds created at watchdog init, drained after each probe. Audit hook: a `write()`-based readability probe must target a pipe/socket, never /dev/null |
